1 Scope & Guiding Principle

What we build now, and the one rule that makes deferring the camera safe.

This plan covers the entire application except the parental camera model. The camera component (liveness + attention verification + attention-quality) is awaiting supervisor sign-off, so we build everything around it first.

The principle that makes this safe The database schema and monitoring endpoints already include the attention_scores table and a camera_enabled flag. So the camera is a drop-in later — the "socket" is built now, the "plug" arrives after approval. No rework, no schema migration when it lands.

What is in Phase 1

  • Full auth with roles (teacher/admin, parent, child) and parent↔child linking
  • Teacher: upload documents, videos, exam papers; create quizzes
  • Child: browse, watch, read, take quizzes, download — with activity tracking
  • Parent: link to child, monitor activity/history (behavioral data only, no camera yet)
  • Engagement scorer (rule-based) and performance predictor (scikit-learn)
  • Three dashboards

What is deferred to Phase 2

  • The camera component (A + B + E) — the attention pipeline that runs in the browser
  • Feeding the real attention score into the engagement index (a placeholder value is used until then)

2 Current System Architecture

The Phase 1 system in one picture — camera shown greyed-out as the deferred module.

The flow is straightforward: a React front-end serves three role-based portals; a FastAPI backend handles auth, LMS logic, quiz grading, and the ML endpoints; Supabase provides the database, authentication, and file storage; and the ML layer is a rule-based engagement scorer feeding a scikit-learn performance predictor.

Why this shape Every layer is independently testable. You can have teachers uploading and children learning (and generating activity data) long before the predictor is trained — the ML layer simply reads whatever data exists.

4 Storage Strategy — Deep Dive

"When a teacher adds a document, where does it go? What about videos?" — thought through properly, because this is the trickiest infra decision in the whole build.

4.1 The core insight: documents and videos are NOT the same problem

A PDF is a few megabytes and gets downloaded once. A video is hundreds of megabytes, needs to stream (start playing before fully downloaded), and — crucially for your research — you need to know how much of it the child actually watched. Treating them the same is the most common mistake here.

4.2 Documents & exam papers → Supabase Storage

  • Store in a Supabase Storage bucket (e.g. materials). It's S3-backed object storage.
  • The database keeps only a reference (the storage path) in learning_materials.storage_path — never the file itself in a DB column.
  • Serve downloads via signed URLs (time-limited links) so only authorised, logged-in students can fetch a file — the file isn't public.
  • Typical size: 1–50 MB. Supabase handles this comfortably.

4.3 Videos → a dedicated video host (not the database, and ideally not plain Supabase Storage)

You can put small videos in Supabase Storage for a prototype, but for anything real, a purpose-built video host is strongly better. Here's the honest comparison:

OptionStreamingWatch analyticsCostVerdict
Supabase StorageProgressive only (no adaptive bitrate)None — you build itStorage + egressOK for prototype / short clips
Cloudflare StreamAdaptive (HLS)Yes, built-in~$5 / 1000 min stored + deliveryBest value for production
MuxAdaptive (HLS)Excellent (watch %, engagement)Pay per minuteBest analytics — helps your ML
Bunny.net StreamAdaptiveBasicVery cheapBudget production option
YouTube unlistedAdaptive (free)Limited, impreciseFreeFree but weak tracking + branding/ToS concerns
Recommendation Prototype: Supabase Storage for both documents and short videos — track watch-time yourself via HTML5 video events. Production: Supabase Storage for documents + Cloudflare Stream or Mux for video. The video host's built-in watch-percentage analytics directly feed your performance-prediction model, so this isn't just an infra choice — it improves your research data.

4.4 How watch-time gets tracked (this is a research signal, so it matters)

The child's video watch-percentage is a feature for Model 1. Two ways to capture it:

  • Frontend-tracked (prototype): the HTML5 video player fires timeupdate events; the frontend periodically POSTs progress to /activity, which writes watch_percent into student_activity.
  • Host-tracked (production): Mux/Cloudflare report watch data via webhooks — more accurate, harder to game.

4.5 The trained ML model file — where does that live?

Easy to overlook: the performance predictor is a saved .pkl file. Store it in a Supabase Storage bucket (e.g. models) or bundled with the API image, and have FastAPI load the active version at startup. Version the filename (predictor_v3.pkl) so you always know which model produced a prediction.

5 Database Schema (Supabase / PostgreSQL)

Paste-ready DDL. The camera table exists now so nothing changes when the camera lands.

Supabase Auth manages auth.users; we add a profiles table for role and details. The full schema is in the accompanying End_to_End_Build_Plan.md; the essentials:

create type user_role as enum ('admin','teacher','parent','child');

create table profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  role user_role not null,
  full_name text not null,
  email text unique,
  grade_level text,
  created_at timestamptz default now()
);

create table parent_child_link (
  id uuid primary key default gen_random_uuid(),
  parent_id uuid references profiles(id) on delete cascade,
  child_id  uuid references profiles(id) on delete cascade,
  relationship text,
  unique (parent_id, child_id)
);

create table learning_materials (
  id uuid primary key default gen_random_uuid(),
  uploaded_by uuid references profiles(id),
  subject_id uuid,
  title text not null,
  type text not null,          -- 'document','video','exam_paper','slide'
  storage_path text not null,  -- Supabase Storage path OR video-host asset id
  duration_seconds int,        -- for videos
  created_at timestamptz default now()
);

create table student_activity (
  id uuid primary key default gen_random_uuid(),
  child_id uuid references profiles(id),
  material_id uuid references learning_materials(id),
  action text not null,        -- 'view','download','video_watch','quiz_submit'
  time_spent_seconds int,
  watch_percent numeric,       -- video watch %  (research signal)
  created_at timestamptz default now()
);

create table monitoring_sessions (
  id uuid primary key default gen_random_uuid(),
  parent_id uuid references profiles(id),
  child_id uuid references profiles(id),
  started_at timestamptz default now(),
  ended_at timestamptz,
  pages_viewed int default 0,
  history_checks int default 0,
  camera_enabled boolean default false   -- Phase 2 socket
);

-- Phase 2 socket: built now, populated later
create table attention_scores (
  id uuid primary key default gen_random_uuid(),
  session_id uuid references monitoring_sessions(id) on delete cascade,
  attention_score numeric,     -- 0..1
  gaze_consistency numeric,
  focus_quality numeric,
  computed_at timestamptz default now()
);
Row-Level Security = your role system's backbone Enable RLS on every table. The single most important policy: a parent can only read rows where child_id is linked to them via parent_child_link. This is enforced by the database itself — even a frontend bug can't leak another family's data.

7 FastAPI Backend Design

The structure, and the golden rule for when to route through the API vs. straight to Supabase.

/backend/app
  main.py                 # FastAPI entry
  /auth/dependencies.py   # verify Supabase JWT + extract role
  /routers
    materials.py          # upload metadata, list, signed-URL download
    quizzes.py            # create, attempt, auto-grade
    activity.py           # log & fetch student activity
    monitoring.py         # start/end session (attention endpoint = Phase 2)
    engagement.py         # engagement index formula
    predictions.py        # run & fetch predictions
    messages.py
  /ml
    predictor.py          # loads predictor_vN.pkl, predicts
    engagement.py         # weighted engagement formula
    train_predictor.py    # offline training script
  /db/supabase_client.py
Golden rule: direct-to-Supabase vs. through-FastAPI Plain data reads/writes (list materials, fetch activity) can go frontend → Supabase directly, protected by RLS. Anything with logic or ML — quiz grading, engagement scoring, predictions, signed-URL generation — goes frontend → FastAPI → Supabase. This keeps the API lean and the app fast.

The auth pattern

  1. Frontend logs in via Supabase Auth → receives a JWT.
  2. Frontend calls FastAPI with Authorization: Bearer <JWT>.
  3. FastAPI verifies the JWT, reads the user's role, and authorises the action.

8 UI / UX Design Plan

Three portals with a shared design language but role-appropriate emphasis.

Design principles

  • Role-appropriate density: the teacher/admin views are data-dense (tables, risk lists); the parent and child views are calm and focused.
  • Support, not surveillance / blame: parent-facing language reads "your child may benefit from support," never "your child is failing." Applies to Phase 2 camera framing too.
  • Mobile-first for parents: in Sri Lanka, parents will most often open this on a phone. The parent portal must work well on a small screen and low bandwidth.
  • Trilingual-ready: plan for Sinhala / Tamil / English from the start (even if v1 ships English) — structure text for easy translation.
  • Explainable predictions: every risk score shows its top 2–3 contributing factors, never a bare number.

9 Phase-by-Phase Implementation Plan

Sequenced so you can start today and drop the camera in later with no rework.

Phase 0 Project Setup Week 1

  • Create Supabase project; set up repo (frontend / backend folders); CI stub.
  • Create Storage buckets: materials, models (+ decide video host).
  • Local dev: Python 3.11+, Node 18+, Docker.
Exit: everyone can run the app skeleton locally against Supabase.

Phase 1 Auth & Roles Weeks 1–2

  • Supabase Auth signup/login; profiles table with role.
  • Parent↔child linking (admin screen).
  • Enable RLS + write the core policies.
Exit: a parent logging in can see only their linked child; roles route to the right portal.

Phase 2 Teacher Content Tools Weeks 2–4

  • Upload documents/exams to Supabase Storage (signed-URL download).
  • Video upload to the chosen host; store reference.
  • Quiz builder (MCQ + short answer).
Exit: a teacher can populate a subject with materials, a video, and a quiz.

Phase 3 Child Learning + Activity Tracking Weeks 4–6

  • Subject browser, material/video viewer, quiz taker, exam downloads.
  • Track time-on-material, watch-%, quiz scores into student_activity.
Exit: a child's session produces real activity data in the DB.

Phase 4 Parent Monitoring (behavioral only) Weeks 6–7

  • Parent views child activity/history; monitoring sessions log behavior.
  • attention_scores table sits ready but unpopulated.
Exit: parent monitoring produces session + behavioral data.

Phase 5 ML: Engagement + Predictor Weeks 7–9

  • Engagement scorer formula (uses a placeholder attention value for now).
  • Train performance predictor offline on simulated + public data; load .pkl in FastAPI; expose predictions with explanations.
Exit: predictions appear in parent/teacher dashboards.

Phase 6 Dashboards Weeks 9–11

  • Teacher risk list, parent overview + prediction, admin analytics.
Exit: all three roles have a usable dashboard.

Phase 7 Camera Component (DEFERRED) After approval

  • MediaPipe A+B+E pipeline in the browser; POST score to the waiting endpoint.
  • Swap the placeholder attention value for the real score. No schema change.
Exit: genuine attention data flows into the engagement index.

10 Deep FAQ — All the Hard Questions

The questions your client and supervisor will ask — answered up front.

QWhen a teacher uploads a document, where does it physically go?

Into a Supabase Storage bucket (object storage, S3-backed). The database stores only the path to it, not the file. Downloads are served through time-limited signed URLs so only logged-in, authorised students can open them.

QWhere do videos get stored, and why not just in the database?

Never in the database — databases are terrible at large binary files. For a prototype, small videos can go in Supabase Storage (progressive playback). For production, use a dedicated video host (Cloudflare Stream or Mux) that streams adaptively and reports how much of each video was watched — which doubles as a research signal for your model.

QHow do we know how much of a video a child actually watched?

In the prototype, the HTML5 video player emits progress events that the frontend periodically sends to the API, writing watch_percent to student_activity. In production, Mux/Cloudflare report this via webhooks — more accurate and harder to fake.

QHow big can uploads be, and what about slow connections?

Supabase supports resumable (TUS) uploads for large files, so an interrupted upload can continue. For Sri Lankan bandwidth realities, adaptive streaming (from a video host) matters a lot — it drops quality gracefully instead of buffering forever.

QHow is a parent stopped from seeing another family's child?

Row-Level Security in Supabase. A single policy restricts parents to rows where child_id is linked to them. This is enforced by the database, not just the app, so a frontend bug can't leak data.

QWhere does the trained ML model file live?

As a versioned .pkl file in a Supabase Storage models bucket (or bundled with the API). FastAPI loads the active version at startup; the version string is stored on every prediction so you always know which model produced a result.

QHow often do predictions run — live on every request?

No. Predictions are computed on a schedule (e.g. nightly or per term) as a batch job, plus on-demand when an admin triggers a re-run. Dashboards read the stored latest prediction — fast, and no model call on every page load.

QDo we need Redis, a message queue, or Kubernetes?

Not for a prototype/pilot. A single API instance + Supabase is plenty. Add caching or a queue only if real usage shows a need — don't pre-build scale you don't have yet.

QDo we need Hugging Face or a GPU?

No. Gaze (Phase 2) uses MediaPipe on the parent's own device; prediction uses scikit-learn on CPU. Neither needs a GPU or Hugging Face. Those only enter the picture for optional future features like a chatbot or auto-quiz-generation.

QWhat happens to the app when the camera isn't approved yet?

Everything works. The engagement scorer uses a neutral placeholder for the attention component; monitoring still logs behavioral data. When the camera is approved, you swap the placeholder for the real score — no schema change, no rework.

QHow do we handle three languages (Sinhala/Tamil/English)?

Structure all UI text through a translation layer (i18n) from day one, even if v1 ships English only. Retro-fitting localisation later is painful; planning for it now is nearly free.

QIs this GDPR/data-protection compliant for minors?

Phase 1 uses simulated + public data, so real-minor risk is low. But build the habits now: consent records, minimal data collection, RLS, and audit trails. The camera phase raises the bar sharply — that's why it goes through the supervisor/ethics review before it's built.

QWhat if the client later wants a mobile app, not just web?

The FastAPI backend is client-agnostic — a React Native or Flutter app can consume the exact same API. No backend changes needed; only a new frontend.

QHow do we back up the data?

Supabase provides automated daily backups (with point-in-time recovery on paid tiers). Learning materials in Storage are durable object storage. Keep the training dataset and model files versioned in the repo/Storage too.

11 My Added Suggestions

Things not in the original brief that are worth deciding on early.

1. Don't show the child their own risk score Telling a 15-year-old they're "high risk of failure" can be genuinely demoralising and self-fulfilling. Keep predictions visible to teacher and parent only; show the child encouraging progress, not a risk label. Worth confirming with the client.
2. Track "time-on-material" carefully — it's gameable A child can open a PDF and walk away. Consider lightweight signals (scroll depth, tab-focus, quiz performance) so "time spent" isn't trivially inflated — the same integrity concern the camera solves on the parent side.
3. Consider the correlation-vs-causation framing in the UI too When you show a parent "more monitoring → better predicted results," phrase it as an association and a nudge, not a promise. Over-claiming in the interface is the kind of thing a supervisor will catch.
4. Build an admin "seed/demo data" tool early Since Phase 1 runs on simulated data, a one-click way to generate realistic demo students/activity makes every later phase (and every client demo) far easier.
5. Log an audit trail from day one Who viewed which child's data and when. Cheap to add now, invaluable for the ethics review and for trust with parents.
